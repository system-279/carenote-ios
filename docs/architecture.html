アプリケーション構造

MVVM + Repository パターンとOutboxによるオフライン対応アーキテクチャ

アプリの構造を「ビル」に例えると

CareNoteアプリは4つの「フロア」で構成されています。それぞれのフロアが異なる役割を担い、下のフロアは上のフロアに依存しません。

4F

画面フロア（Features）

ユーザーが直接触れる画面。録音ボタン、一覧表示、設定画面などがここにあります。

3F

業務処理室（Services）

録音の開始・停止、クラウドへのアップロード、AI文字起こしの依頼など、実際の「仕事」を行うフロア。

2F

データ管理室（Repositories）

端末内のデータベースへのデータの保存・取得を管理。「何をどこに保管するか」を決める窓口です。

1F

データ保管庫（Models）

録音データ、クライアント情報、テンプレートなどのデータ構造の定義。「データの形」を決めています。

オフライン対応の仕組み

CareNoteは通信が不安定な介護現場でも安心して使えるよう、Outbox（送信待ちトレイ）パターンを採用しています。

📡
録音データはまず端末に保存されます。インターネットに接続されると自動的にクラウドへアップロードされ、AI文字起こしが実行されます。通信が途切れても、接続が回復すれば自動で再開します（最大3回再試行）。

MVVM + Repository Pattern

ComponentRoleConvention
ViewUI描画・ユーザー操作SwiftUI、ロジック不含、@Query 直接使用禁止
ViewModel画面状態管理・Service呼び出し@Observable マクロ（ObservableObject 不使用）
Serviceビジネスロジックactor または struct、async/await
RepositorySwiftData CRUD抽象化actor、ModelContext のスレッド安全性保証
Modelデータ構造定義@Model (SwiftData)、Codable (Firestore)
⚠
tenantId のハードコーディング禁止。すべてのService/RepositoryはパラメータでtenantIdを受け取ります。Firebase Auth custom claimから取得した値をViewModelレイヤーで管理し、下位レイヤーに渡します。

Outbox Pattern - State Machine
